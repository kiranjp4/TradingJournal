# ISO 26262 — 90-Day Practical Learning Plan

**Goal:** Read the standard systematically **and** build real artifacts (HARA, safety goals, FMEA, traceability, safety case sketch).

**Pace:** ~30–45 min/day, Mon–Fri + weekly review on Saturday.

---

## Phase 1: Foundations (Days 1–14)

| Day | Read | Practical exercise | Output file |
|-----|------|-------------------|-------------|
| 1 | 26262-1: Scope, normative references | List 5 systems in your life/work that need FuSa; pick one "learning item" | `projects/learning-item.md` |
| 2 | 26262-1: Item, element, fault, failure | Draw a block diagram of your learning item (3–5 elements) | `projects/learning-item.md` |
| 3 | 26262-1: ASIL, safety goal, safety requirement | Write 1-sentence definitions for 10 terms in glossary | `glossary/terms.md` |
| 4 | 26262-2: Overall safety lifecycle | Draw lifecycle phases on paper; photo or mermaid in notes | `notes/26262/part-02/lifecycle.md` |
| 5 | 26262-2: Roles (PM, SM, architect, etc.) | Assign roles for a fictional project team of 5 | `notes/26262/part-02/roles.md` |
| 6 | Review | Re-read Days 1–5 logs; fix weak glossary entries | `glossary/terms.md` |
| 7 | Rest / catch-up | | |
| 8 | 26262-3: Item definition | Write item definition for learning item (purpose, boundaries, interfaces) | `projects/learning-item.md` |
| 9 | 26262-3: HARA — operational situations | List 10 operational situations for your item | `projects/mini-hara.md` |
| 10 | 26262-3: HARA — hazard identification | Identify 5 hazards with malfunction references | `projects/mini-hara.md` |
| 11 | 26262-3: S/E/C rating and ASIL | Rate 5 hazards; determine ASIL (use table in standard) | `projects/mini-hara.md` |
| 12 | 26262-3: Safety goals | Write 2 safety goals with ASIL; add safe states | `projects/mini-hara.md` |
| 13 | Review | Trace: situation → hazard → ASIL → safety goal | `logs/` |
| 14 | Rest / catch-up | | |

## Phase 2: Concept → System (Days 15–35)

| Day | Read | Practical exercise |
|-----|------|-------------------|
| 15–17 | 26262-3: FSC, assumptions, dependencies | Draft Functional Safety Concept for 1 safety goal |
| 18–20 | 26262-4: Technical safety requirements | Derive 3 TSRs from one safety goal |
| 21–23 | 26262-4: System architectural design | Allocate TSRs to elements in block diagram |
| 24–26 | 26262-9: FMEA introduction | Start DFMEA on one element (5 rows min) |
| 27–29 | 26262-4: System integration & verification | Write 2 verification test descriptions for TSRs |
| 30–32 | 26262-2: Safety case structure | Outline safety case: goals, arguments, evidence |
| 33–35 | Review phase | Complete traceability table hazard → goal → TSR → test |

## Phase 3: Hardware OR Software track (Days 36–60)

Pick **one track** based on your job:

### Software track (26262-6)

| Days | Topic | Exercise |
|------|-------|----------|
| 36–38 | SW safety lifecycle | Map your SW phases to lifecycle table |
| 39–42 | SW safety requirements | Derive 3 SW safety requirements from TSR |
| 43–46 | Architectural design | Simple component diagram + safety mechanisms |
| 47–50 | Unit design & implementation | Write pseudo-code for one safety mechanism |
| 51–54 | SW unit/integration testing | Draft test cases for one SW requirement |
| 55–58 | Tool qualification / coding guidelines | Review MISRA or company coding rules; note 5 rules |
| 59–60 | Review | Update safety case with SW evidence |

### Hardware track (26262-5)

| Days | Topic | Exercise |
|------|-------|----------|
| 36–38 | HW safety lifecycle | Map HW phases to lifecycle |
| 39–42 | HW safety requirements | Derive HW requirements from TSR |
| 43–46 | HW design & FMEDA intro | FMEDA on one circuit block (simplified) |
| 47–50 | HW metrics (SPFM, LFM) | Calculate metrics for toy example |
| 51–54 | HW integration & verification | Draft HW test for one requirement |
| 55–58 | HW element analysis | Document failure modes for one component |
| 59–60 | Review | Update safety case with HW evidence |

## Weekly Saturday review (15 min)

1. Read last 5 daily logs.
2. Update `glossary/terms.md` (minimum 3 terms).
3. Check off `PROGRESS.md`.
4. Commit: `git commit -m "FuSa Week N review"`.

## Suggested learning items (pick one)

| Item | Why it's good for practice |
|------|---------------------------|
| Windshield wiper system | Clear hazards, familiar, good HARA practice |
| Exterior lighting ( brake lamp ) | Simple, ASIL varies, good for traceability |
| Electric parking brake | Medium complexity, HW+SW |
| Battery management (monitoring only) | Real automotive, good for SW track |
| Industrial emergency stop (IEC 61508) | If not automotive — use IEC track instead |

## After 90 days

- Re-read weak parts from gap analysis.
- Start **ISO 21448 (SOTIF)** if ADAS/autonomy is relevant.
- Join internal FuSa reviews at work — use your mini project language.
